# Tools Catalog

自分が作ってきたツールの棚卸し。3D の領域とは別に「作ってきた実体」を並べる。

作った理由 = 違和感の言語化。表面のツール名の下に「本当に解きたかったこと」を書く。

---

## 業務系（片江店 / ネクスト城南）

### katae-yugai-hyo — 片江店 油外実績表
- 何: 油外実績のカレンダー入力＋目標管理 Web アプリ（GAS）
- 表向きの動機: 実績を毎日ラクに入力し、月次目標を可視化する
- 下にある違和感: 「達成/未達」が数字だけだと現場が動かない。**日次で目に入る色**にしないと意味がない（アクセント `#D7263D` に赤を選んだ理由）
- 続きの問い: この赤は「圧」になっていないか？「圧」と「意識」の境界を分ける工夫は？

### yugai-kanri — 油外管理ツール（バインド版）
- 何: 上と同系統だがスプシバインド型 GAS Web アプリ
- 下にある違和感: **standalone と bound の二重管理**が発生している ← ここ自体が違和感
- 続きの問い: なぜ2本並存しているのか？片方に寄せる決断がなぜ保留されているのか

### shift-kanri-tool — 企画部シフト管理
- 何: 企画部3名（自分・田口・鈴木）のシフト＋会議/SNS予定管理、カレンダー同期
- 下にある違和感: メンバー個人のカレンダーに勝手には反映されない → **同期を手動ボタンにした**（自動トリガーOFF）
- 「他人を勝手に動かさない」設計思想が出ている
- 続きの問い: この配慮は全ツール共通か？「勝手にやる」と「頼まれて動く」の線引きが自分の中にある

### weekly-mtg-tool — 週次MTGツール
- 何: 議題・ネクストアクション管理＋NAをGoogleカレンダーへ自動登録
- 下にある違和感: **「議題を出した人・解決できていない議題」が消えて忘れ去られる**
- 続きの問い: 忘却との戦い。「持ち越し」機能がある = 過去の未処理を可視化したかった

### kikaku-os-web — 企画OS Web版（Firebase）
- 何: 企画部のWeb運用インフラ（Firestore + Hosting）
- 下にある違和感: GASでは重い/共有が難しい機能を、Web/Firestoreに逃したかった
- 続きの問い: なぜGASで足りない？

### kikaku-suishin-os — 企画推進OS
- 何: 巨大GAS（Code.gs 176KB、Flat7Line 系ファイル多数）＝ LINE配信+複数機能の統合OS
- 下にある違和感: 個別ツールを増やしすぎた → **1つに束ねたい**という反動が出た
- 続きの問い: 「1つに束ねたい」と「独立させたい」の間で自分はどこに立っているのか

## メタ観察

上を眺めてわかること:

1. **「他人に配れる形」への異常なこだわり** — 単独ツールでも必ずマスタ・共有・カレンダー連携を組み込む
2. **AIに任せる/任せない の線引きが明確** — nousha-blog-tool は AI生成を排した
3. **UI/UX の温度感を握りたい** — 色・境界・分離を必ず自分で決めている
4. **「見て分かる」を最優先** — カレンダー、レポート、Hook Score、3D、全部「一目で状態が分かる」形
5. **並存を許容してから統合する順** — 保坂ルーティン(PWA/GAS)、油外(standalone/bound)を両方作って様子を見ている
6. **GAS を基盤に選び続けている** — 15/17ツール中10本以上がGAS = 認証済みGoogle環境で完結する安心感
